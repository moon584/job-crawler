### 2.1 架构设计
为了支持各类招聘官网（包括需要解密、定制接口甚至直接解析HTML的官网），爬虫采用了基于 **Provider 工厂模式** 的架构。目前分为四大类爬虫驱动：
1. **ConfigProvider (`ConfigDrivenProvider`)**
   - **适用场景**：接口规范，返回标准 JSON 且结构简单的 API（如腾讯）。
   - **实现机制**：通过 `rules/company.json` 中的海量 JSON 路径配置进行字段映射，代码完全通用。
2. **CustomApiProvider (定制化 API 爬虫, 如 `MeituanProvider`)**
   - **适用场景**：接口返回 JSON，但需要复杂的签名（Token）、加密解密、特定分页结构或动态请求参数重组（如美团）。
   - **实现机制**：继承自 `BaseProvider`，在 `crawler/providers/impl/` 下使用 Python 原生代码重写 `fetch_posts`、`fetch_detail` 或相关解析逻辑。
3. **HtmlProvider (`HtmlProvider`)**
   - **适用场景**：服务端渲染的静态网页（SSR），不返回 JSON 而返回 HTML 代码。
   - **实现机制**：底层自动请求 HTML 文本，子类只需实现 `parse_list_html` 和 `parse_detail_html` 方法，利用 BeautifulSoup 等工具编写 XPath/CSS 提取逻辑。
4. **DynamicProvider (预留方案)**
   - **适用场景**：高度动态、反爬严格的单页应用（SPA）。
   - **实现机制**：未来可通过封装 Playwright / Selenium 接管底层网络请求。

### 2.2 动态认证刷新机制 (AuthError)
应对类似美团这类需要登录（Cookie/Token）且容易过期的爬虫：
- 当网络请求或解析返回特定的状态码（如 `status=401`）时，Provider 会抛出 `AuthError` 异常。
- 框架会捕获该异常，并自动调用该 Provider 重写的 `refresh_auth()` 方法。
- 若 `refresh_auth()` 执行成功（如重新模拟登录拿到了新 Cookie），框架将自动携带新凭据重试刚刚失败的接口，确保抓取任务不中断。
### 2.4 安全的软删与清理 (is_deleted)
旧版爬虫在完成快爬对比或慢爬时，会直接 `DELETE` 那些没有再次被抓到的过期帖子，这在自动分类模式下非常容易因为一次局部报错引发全公司数据被误删。

当前重构后采用“心跳时间戳（Timestamp Heartbeat）”模式：
1. **事后软删标记**：当 `_crawl_start_time` 记录下当前次爬虫启动时，所有本次爬虫中碰到的职位，都会被更新其时间戳 `crawled_at` 至本次爬虫的时间。在整家公司（或整个分类）爬取**结束后**，系统对比那些 `crawled_at` 仍然比启动时间旧的记录，即可判定该记录已经下线了。随后框架统一把下线记录标记为 `is_deleted=1`，避免在爬取中间或接口报错中断时误删数据。如果再次抓到这个网址，会自动恢复为 `0`（“复活”）。
2. **人工异步执行硬删**：框架提供了 `clean_deleted_jobs.py`，你可以在确信本次全量爬取没有任何网络拦截或阻断后，手动运行这个脚本安全地物理删除那些下架帖子。

6.其他要求：

- **统一心跳更新机制（原快爬/慢爬合并）**：每次抓取都是**增量更新**，并引入字段级对比：当抓取到的列表（List API）数据与数据库中已有记录匹配时（基于 `job_url` 或 `PostId`），如果配置为不需要详情即更新，则直接触碰更新 `crawled_at` 时间戳并标记 `is_deleted=0`；否则将进一步请求详情接口（Detail API）并对比数据库中的标题、薪资、要求等所有字段。当岗位没有任何变化时，只更新 `crawled_at`；有任何变化时才会执行 `UPDATE`，并仅覆盖变动的字段。

- 设置合理的延迟（如 0.5~1 秒/请求），避免给服务器造成压力。

- 增设异常处理与重试

- **记录爬取日志**：打印成功/失败数量，便于监控。
